# NightTable вЂ” Architecture

## Stack
- Framework: Next.js 14 with App Router
- Language: TypeScript (strict)
- Styling: Tailwind CSS (custom design system, dark only)
- Database: PostgreSQL via Supabase
- Auth: Supabase Auth (5 roles)
- Payments: Stripe (Payment Intents, Subscriptions, Webhooks)
- AI: OpenAI GPT-4o (reservation chatbot, streaming)
- SMS: Twilio
- Email: Resend
- Floor plan: Konva.js (interactive canvas)
- Validation: Zod
- Deployment: Vercel (with cron jobs)

## Data Flow Patterns

### Reservation flow
Client selects table в†’ calculateDynamicPrice() в†’ Stripe Payment Intent created
в†’ Client pays в†’ Stripe webhook fires в†’ reservation.status = 'confirmed'
в†’ SMS via Twilio + Email via Resend в†’ Promoter commission created if promo_code_used

### Promoter tracking flow
Promoter shares /reserve?promo=CODE в†’ Client clicks в†’ cookie set (48h)
в†’ Client books в†’ Server Action reads cookie в†’ promoter_clicks logged
в†’ commission created on webhook в†’ monthly payout

### Dynamic pricing

в†’ Stored in event_tables.dynamic_price
в†’ Recalculated on each new reservation in the same event

## Key Design Decisions
- PWA over native app: no App Store friction, works on all phones
- Server Components by default: SEO, performance, smaller JS bundle
- Server Actions for mutations: no separate API layer needed for internal calls
- Supabase RLS: security enforced at database level, not just middleware
- Stripe webhooks as source of truth: never trust client-side payment confirmation
